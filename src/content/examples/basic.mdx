## Why this structure for one feature?

You're testing one form. Why four files?

- **`components.ts`** - when the login form's HTML changes (and it will), you change one file. The selector strategy stays in one place.
- **`data.ts`** - when QA needs a new test user, they add a fixture. No git diff in `test.ts` for data churn.
- **`actions.ts`** - `login(creds)` is reusable. The next feature that needs an authed session imports `LoginActions` and calls `login(VALID_USER)`. No copy-paste.
- **`test.ts`** - assertions are clearly demarcated. If a test fails, you know exactly which scenario broke; the assertion is right there.

<Callout type="tip" title="When to break out into more files">
  When a feature has 3+ user flows (e.g., "login", "logout", "password reset" all in `auth/`), keep them in **one** Actions class with multiple methods. Don't split until you're hitting 500+ lines.
</Callout>
